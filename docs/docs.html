Getting started

One static binary. The installer verifies the release tarball against the checksums.txt published with each release; or build from a checkout with cargo build --release --features sandbox (release binaries ship with sandbox support compiled in; a bare cargo build does not) and put it on PATH with agentstack self link.

1
agentstack init — scans your machine for the CLIs you use, imports their MCP servers, and writes one reviewed manifest at .agentstack/agentstack.toml. Inline tokens are lifted to ${REF} placeholders.
2
agentstack apply --write — renders the manifest back into every CLI's native config. Edit once, applied everywhere.
3
agentstack doctor — health and drift checks; every warning names its exact fix. Undo any recorded write with restore --last --write.

Concepts

manifest
One file — .agentstack/agentstack.toml — listing everything your tools may run: MCP servers, skills, instructions, settings, hooks. It only ever holds ${REF} secret placeholders, never real values.
lockfile
agentstack.lock pins the manifest's resolved contents to SHA-256 digests. Same inputs reproduce; any change is visible. Part of what you consent to when you trust a project.
adapter
The per-CLI compiler: one YAML descriptor per CLI, embedded in the binary, turning the manifest into native config. Thirteen today: Claude Code, Claude Desktop, Codex, Cursor, Windsurf, Gemini CLI, VS Code, Copilot CLI, OpenCode, Antigravity, Junie, Kiro, Pi.
gateway
The in-process broker. Calls route through it instead of straight to MCP servers, so policy is checked and every call is logged. Only brokered calls are policy-checked and audited.
trust
Your approval that a project may auto-load on this machine, pinned to a consent digest. Until you trust, a clone is inert. Change any pinned byte and it drops back to untrusted.
policy
Rules fencing tools, egress, secrets and filesystem paths. Effective policy = machine ∩ project — a repo can only narrow your rules, never loosen them.
profile
A named subset of the manifest ("backend", "design") activated together with use <profile>. A manifest with no profiles activates its whole inline set.
posture
The per-run label on the run banner — HOST/ADVISORY, HOST/PROTECTED, SANDBOX/PROXIED or LOCKDOWN/ENFORCED — what the effective policy is actually enforced by.

How-tos

trust
Trust a cloned repo
Clone, open — it's inert. Run agentstack trust . to review exactly what it declares (servers, contacts, secrets, skills), then approve. A later git pull that changes pinned bytes drops it back to inert.
servers
Add an MCP server
agentstack search <name> looks across your library, the bundled catalog and the official MCP Registry; add from <id> lifts its secrets and renders it to every CLI at once.
skills
Add a skill
add skill owner/repo installs from any repo, pinned in the lockfile. Try one first without writing anything: agentstack try <skill>.
audit
See what happened
report run <id> replays a run's flight recording; report calls shows the global audit log (argument digests only, never values); dashboard serves it all read-only in a browser.
team
Share one setup with your team
Commit .agentstack/ (manifest + lockfile) — intent, never credentials. Each teammate runs secret set for their own values, then apply --write. A maintainer can sign the lockfile (detached ed25519); teammates verify --pubkey before relying on it.
lockdown
Lock down a run
Preview with --plan (walks every gate, launches nothing). Then climb: --locked (no Docker, frozen surface) → --sandbox (container + proxied egress, direct route still open) → --sandbox --lockdown (internal-only network, sole peer the egress sidecar).
undo
Undo anything
restore lists every undoable recorded write; --last --write reverts the most recent. Non-write actions have their own verb — see the Undo section below.

Which mode do I need?

Two decisions: how much protection, and where rendered files live. Find the row that sounds like you — the rows stack, so most people combine several. Preview any run first with --plan: it walks every gate, launches nothing, needs no Docker.

You are…RunWhat it actually does
just syncing config across your CLIsinit → apply --writeCopies your reviewed config into each CLI. No runtime check — nothing is blocked once an agent is running.
worried about rm -rf or .env accidentsguard installCooperative: catches an agent's accidents through each CLI's own hook. Not a determined attacker.
cloning repos you didn't writegateway connect → trust .A repo's servers, skills and secrets stay inert until you trust it. Trust gates loading — it does not sandbox the code.
launching a frozen, verified surface — no Dockerrun <cli> --lockedFail-closed trust and pin checks before launch, then a frozen surface. HOST / PROTECTED — the agent still runs as you.
running sensitive work that must not leakrun --sandbox --lockdownContainer on an internal-only network whose sole peer is the egress sidecar. LOCKDOWN / ENFORCED · NO DIRECT ROUTE. Needs Docker.

Delivery modes — where rendered files live

You always commit the intent (agentstack.toml + agentstack.lock). Where the rendered artifacts — .mcp.json, .claude/skills/, the compiled CLAUDE.md / AGENTS.md — live is a per-project choice, presented as an arrow-key pick during interactive init:

default
static
Artifacts on disk, kept out of git by a managed .gitignore block that tracks only files agentstack actually wrote. Pass --no-gitignore to commit them instead.
quiet trees
clean-at-rest
agentstack lock pins name refs without rendering, so git status stays silent. A profile arrives via session start / run and reverts on exit.
nothing rendered
zero-files
gateway connect registers the gateway once per CLI; every trusted repo serves its own stack live. A lease fences one MCP connection to a profile with no cleanup contract.

Recommendation: prefer the zero-files lease path for interactive work when the CLI supports MCP; use static or clean-at-rest when the CLI must read native skill or instruction files from disk. A lease is a capability fence, not a sandbox — add --sandbox --lockdown when the agent process itself needs isolation.

Agent-operable — agentstack as an MCP server

AgentStack itself runs as an MCP server over stdio (agentstack mcp), so the agent can discover and propose capabilities — proposals write the manifest only, with commit-safe ${REF}s; nothing executes until a human runs apply. Register once per CLI with gateway connect --all --write.

Discover & vet
agentstack_search finds capabilities across catalog and library; agentstack_explain, _doctor and _diff let the agent inspect state before relying on it.
Propose, never apply
agentstack_add_from, _add_server, _add_skill and _create_profile write the manifest only — the agent proposes, a human reviews and runs apply.
Compact tool surface
Upstream tools collapse behind tools_search instead of flooding tools/list, so tool context stays bounded however many servers you add. --transparent instead exposes every policy-filtered tool as server__tool for any standard MCP client.
Leases & code mode
agentstack_lease_open fences one connection to a profile (zero-files counterpart of a session). tools_bindings hands the agent a typed, secret-free TypeScript client so one small program can call several upstream tools — real calls brokered over a loopback, token-gated endpoint.

Undo anything

restore is the single undo verb for writes — every write is recorded before it lands. Five other actions have their own verb because they are not file writes:

To undo…Run
a recorded write (apply / use / session / settings / hooks)restore <id> --write
a gateway registrationgateway disconnect <cli> --write
the destructive-command guardguard uninstall
trust for a repotrust --revoke
an active sessionsession end
a server or skill in the manifestremove <name> --write

Limit: restore reverts agentstack's own recorded config writes, not side effects a tool already had — a file a server deleted is not brought back.

Security model — the honest version

Trust is consent, not a safety claim
Trust pins a SHA-256 consent digest of manifest + overlay + lockfile and gates whether declared servers may load. It does not vouch the code is safe and does not confine a running server — vet a capability first with agentstack explain <name>.
Guard is cooperative
Hooks catch an agent's accidents, not a determined attacker. A CLI that ignores its own hooks bypasses guard entirely — kernel-level confinement is what run --lockdown is for.
Even lockdown has bounds
Egress rules bound where data can go; they cannot read intent. Traffic to a host you approved — including the model API — is still allowed. Postures from weakest to strongest: HOST/ADVISORY → HOST/PROTECTED → SANDBOX/PROXIED → LOCKDOWN/ENFORCED.
