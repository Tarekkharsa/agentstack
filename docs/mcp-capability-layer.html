Beyond MCP connectors.

What changes when MCP becomes a capability layer—and why agentstack is unusually well positioned to make generated capabilities trusted, portable, constrained, and observable.

Inspired by Adam’s “I was wrong about MCPs.” Read the original article on X ↗

Verdict: strategically useful

The article validates agentstack’s direction. Adopt its execution model; do not expand agentstack into a personal-assistant product.

HIGH RELEVANCE
01 · The conceptual shift

From a catalog of tools to an environment for making capabilities.

Traditional MCP servers translate one service into many predefined tools. The alternative keeps the exposed interface small, discovers capabilities only when needed, and lets an agent compose them into tested code that can be executed again.

01

Discovery over disclosure

Load one relevant schema when needed instead of billing every upstream tool description on every turn.

02

Generation, then determinism

Use the model to write and test a program once; reuse deterministic code instead of repeating a fragile reasoning chain.

03

Primitives over workflows

Provide small capabilities with clear contracts so agents can compose new behavior without a bespoke integration for every task.

04

Control is part of execution

Generated capabilities need least privilege, isolation, approvals, verification, observability, and rollback—not merely a code runner.

03 · Recommendation

Ship a trusted tools_execute.

The gated first milestone is now implemented: a minimal sandboxed TypeScript executor behind the agentstack gateway, disabled unless the machine owner opts in. A focused implementation review and regression-hardening pass completed on 2026-07-13; it remains experimental pending supply-chain attestations and longer-running soak evidence. 3 · Govern

Trust, policy, sandbox, secret scoping, verification, and recording apply.

  1. Start ephemeral.

    Accept one TypeScript program per call. No package persistence, scheduling, or background services in the first version.

  2. Make authority explicit.

    The request names allowed bindings. The effective grant is the intersection of those names with trusted bundle and machine policy.

  3. Remove ambient power.

    No host filesystem, inherited environment, unrestricted network, or raw credential access. MCP calls remain gateway-brokered.

  4. Bound every run.

    Enforce wall-clock, CPU/memory, output-size, call-count, and response-size limits; cancellation must kill the entire process tree.

  5. Record enough to reproduce.

    Log code digest, runtime version, granted bindings, tool calls, policy decisions, timing, and terminal outcome—without logging secrets.

  6. Add reviewed packages second.

    Promote successful programs into content-addressed library entries that re-enter the existing lock, review, trust, signing, and distribution lifecycle.

04 · Product boundary

Be the control plane beneath assistants.

Keep agentstack narrow.

Kody-like products can own personal memory, OAuth onboarding, schedules, user-facing apps, and durable assistant state. Agentstack should own the portable security contract: which generated capability may run, with which tools and secrets, inside which boundary, and with what evidence afterward.
