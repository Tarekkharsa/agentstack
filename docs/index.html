Cloning a repo shouldn't hand
your agent to a stranger.

One reviewed file defines what your AI coding tools — Claude Code, Codex, Cursor, and ten more — may run. A repo you clone can't activate any of it until you approve it. Sensitive runs go in a container whose only way out is an egress firewall — and every call routed through AgentStack lands in the audit log.

01 · You juggle several CLIs$ agentstack apply

One manifest keeps them all in sync

Claude Code at work, Cursor for edits, Codex in CI — and every MCP (Model Context Protocol — the plugin standard agent CLIs use for tools) server configured three different ways. Define it once; agentstack apply renders each CLI's native config, with secrets left as per-machine references.

The manifest, in detail

  env = { GITHUB_PERSONAL_ACCESS_TOKEN = "${GITHUB_TOKEN}" }
 
$ agentstack apply --write

  Codex CLI     + .codex/config.toml       ✓ wrote 1 server
  Cursor        + .cursor/mcp.json         ✓ wrote 1 server

  ✓ same server, three native shapes — the manifest never holds the token

03 · You clone repos you didn't write$ agentstack trust .

They stay inert until you consent

The repo you just cloned declares MCP servers you've never read. Nothing spawns and no secret resolves until you review what it declares and trust it — and any later edit to the pinned surface re-gates it.

How consent works
clone → inert → trust → firewalled → logged
$ agentstack gateway connect --all --write     # once: the gateway, in every CLI
$ git clone some-repo && cd some-repo  # declares servers you've never seen
  agent session starts…
  ✗ untrusted — no servers spawned, no secrets resolved
 
$ agentstack trust .                   # you review, then consent
  ▶ demo: runs `python3 ./server.py`
  ✓ trusted at sha256:945b4b…
 
  agent → demo.echo        ✓ ok      brokered, logged
  agent → demo.delete_all  ✗ denied  rule "!delete_*" — YOUR machine policy
 
  # both brokered outcomes are in the audit log
04 · Some work must not leak$ agentstack run --lockdown

Lockdown: no route out but the proxy

The trust gate decides what an agent may do; lockdown decides what it can. For runs that touch sensitive code or data: --lockdown puts the agent in a container whose only network peer is the egress firewall. Ignoring the proxy reaches nothing; a denied host is refused — and recorded.

The enforcement story
no route out · denied host blocked · on the record
$ agentstack run --lockdown shtest -- -c 'unset HTTPS_PROXY; curl example.com'
  posture: LOCKDOWN / ENFORCED · NO DIRECT ROUTE
  🔒 no host route, no internet — the only peer is the egress sidecar
  BLOCKED        # ignoring the proxy reaches nothing
 
$ agentstack run --lockdown shtest -- -c 'curl https://blocked.invalid/steal'
  ✗ refused at the sidecar — and recorded
 
$ agentstack report run r-0859dcee73
  Posture   LOCKDOWN / ENFORCED · NO DIRECT ROUTE
    ✗ shtest → blocked.invalid   denied by rule "!blocked.invalid" (machine policy)

Why this exists

Your agents run whatever your repos tell them to.

Every skill, MCP server, and agent config you adopt is unreviewed code plus instructions, wired into a process that holds your credentials, your shell, and the network. Installing one is npm install with an agent attached — except with no lockfile, no review gate, and no record of what it did. Three gaps follow from that:

One clone away

A repo can declare MCP servers, and the moment an agent session starts, they want to run — commands you never read, with your keychain in reach.
→ Here, a clone is inert until you approve its consent surface. Any manifest, local-overlay, or lockfile edit re-gates it; referenced local code must still be reviewed separately.

One prompt away

An injected instruction can steer a well-behaved agent into using a legitimate tool against you — deleting, leaking, phoning home.
→ Here, your machine policy — which no valid repo policy can loosen — fences tools, secrets, and egress; --lockdown removes the agent's direct network route while approved traffic can still pass through the enforcing proxy.

Invisible afterwards

When something does go wrong, most setups can't answer the first question a reviewer asks: what did the agent actually call?
→ Here, every brokered call and denial lands in the call audit; gateway-routed sandbox runs also record secret-reference access in their per-run evidence.

Honestly scoped

No tool can make agent use risk-free: hosts you allow stay allowed, and payloads aren't inspected.
→ That's why every mode carries an honest posture label — and if you run one agent with one hand-managed server, you may not need this yet. The moment capabilities come from repos you didn't write, you do.

What's enforced where

"Trusted" is not "confined." Here's the difference.

Trusting a repo approves what it may load. How strongly a run is confined is a separate choice — how you launch it. Every run prints a posture label that says exactly how strongly its rules are enforced:

host — HOST / ADVISORY
No container. The gateway still brokers tool calls, but nothing confines the process's own egress or filesystem.
gateway — GATEWAY / BROKERED
Every MCP call routes through the in-process broker: firewalled and audited, but the process runs on the bare machine.
--sandbox — SANDBOX / PROXIED · DIRECT ROUTE OPEN
Proxied HTTPS egress is checked against your compiled policy — but the container keeps an ordinary bridge, so a proxy-ignoring process can still dial out. --lockdown closes that route.
--lockdown — ENFORCED · NO DIRECT ROUTE
Container on an internal-only network whose sole peer is the egress sidecar. Enforced and topologically confined.

Read every enforced as a ceiling: the disallowed action is prevented at runtime — never that an allowed one is safe. The full six-dimension matrix, per-cell reasoning checked against the source: the enforcement matrix.

Portable config

The same reviewed file, rendered into every CLI

Prefer native config files — instead of, or as well as, the live gateway? The same manifest renders them. And you don't start from a blank page: init reads the agent config already on your machine and turns it into a manifest you can review, commit, and share. (Nothing installed yet? init writes a commented starter manifest instead — no dead ends on a clean machine.)

Serves — and renders native config for —

Claude Code Codex CLI Cursor Windsurf Gemini CLI VS Code GitHub Copilot CLI Claude Desktop OpenCode Antigravity Junie Kiro Pi
  1. agentstack init

    Imports the servers and skills you already have; inline tokens become ${REF}s, stored in a gitignored .env or your OS keychain — never the manifest.

  2. agentstack apply

    Preview each CLI's config diff, confirm to write. Writes are atomic, backed up, and tracked — restore undoes them.

  3. agentstack use <profile> --write

    Activate a profile: its servers and skills, materialized for every CLI. The interactive init wizard runs this step for you.

  4. agentstack doctor

    Any time something feels off. Every warning names the exact fix command.

The manifest

One file, reviewed like code

version = 1

[servers.github]
type = "http"
url = "https://api.githubcopilot.com/mcp/"
headers = { Authorization = "Bearer ${GH_PAT}" } # resolved per machine, never stored

[profiles.backend]
servers = ["github", "kibana"]
skills  = ["sql-review"]               # resolves from your central library

[targets]
default = ["claude-code", "codex"]

Secrets stay ${REF} placeholders that resolve per machine; an unresolved one blocks the write, so a value never leaks into live config. A lockfile pins every skill and server by digest, and install --locked reproduces the setup in CI.

Rendered files

Pick where artifacts live

You always commit the intent (manifest + lock); the rendered artifacts (.mcp.json, .claude/skills/) are a per-project choice:

static default
Rendered configs sit on disk, gitignored by a managed block.
clean-at-rest
Nothing persists between sessions — injected on launch, reverted on exit.
zero files
Nothing rendered at all; the gateway serves each repo's servers live, behind the trust gate.

Teams & CI

Clone, apply, done

git clone <repo>
agentstack secret set GH_PAT     # local only; never committed
agentstack apply --write
agentstack doctor                # verify the wiring

In CI, the trust gate is two commands — or the one-line GitHub Action:

agentstack install --locked   # fail if sources drifted from the pinned lock
agentstack doctor --ci        # fail on errors, drift, policy, unsafe content

The step-by-step versions, with limits and the optional sign/verify provenance path: team setup · use in CI.
