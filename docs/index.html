The mental model.

Three ownership rules explain everything on this page. Centralize capabilities when you are ready; never centralize provider-private state.

AgentStack owns definitions

Skills and MCP server definitions live once under ~/.agentstack/lib and are indexed in library.toml.

Projects own selection

A project profile names what it needs. The project does not need to copy the skill or server definition.

Providers receive views

Provider folders get symlinks or generated config entries only. AgentStack never owns the whole provider folder.

Why you may still see files inside some projects: existing projects can keep local skills, inline MCP definitions, and generated provider files. That is supported. The clean central model is shown by agentstack-test/projects/central-demo, which contains only an agentstack.toml manifest.

Choose your path.

Start from where your project is today. You do not have to migrate everything at once.

Starting point What to do Result
Existing project with [skills.*] and [servers.*] Keep it working. Run agentstack doctor, then centralize one skill or server at a time. Backwards compatible local project, with optional central items.
Existing provider folders with scattered skills Run agentstack consolidate, review the preview, then agentstack consolidate --write. Skills move into ~/.agentstack/lib/skills; provider skill dirs become symlinks/views.
New project Add capabilities to the central library first, then reference names in profiles. Project contains selection only: servers = ["kibana"], skills = ["sql-review"].
Team or second machine Use locks, doctor, and explain to verify resolved digests and drift. Same names resolve to the same content or fail loudly.

The three modes: where rendered files live.

Independently of the path above, each project picks one of three artifact modes. This decides what a git status ever shows.

Mode How Trade-off
Static (default) use <profile> --scope project --write — artifacts on disk, auto-kept out of git via a managed .gitignore block (--no-gitignore to commit them). Works with harnesses launched any way; generated files exist on disk.
Clean-at-rest Empty [profiles.off] + use off --scope project --write once; then agentstack run <cli> --profile <p> or session start/end inject and revert per session (pruning-to-zero deletes even the empty config file). Nothing generated exists between sessions — but a harness launched directly sees no project servers/skills while at rest. One use flips back to static.
Zero files (MCP) Register agentstack mcp globally; the agent lists and loads library skills at runtime via agentstack_list_loadable / agentstack_load (session-fenced, loads logged with reasons). No files in the repo at all; skill triggering is model-driven rather than automatic frontmatter matching.

What goes where.

The same capability can pass through fixtures, central library, project manifest, generated provider view, and lockfile. Each has a different job.

Clean central-demo source tree

agentstack-test/ fixtures/ central-library/ kibana.toml sql-review/SKILL.md projects/ central-demo/ agentstack.toml runtime/ # gitignored ashome/lib/ # real central library after script runs home/.claude.json # generated provider view

What should not be treated as source

projects/*/.mcp.json # generated project MCP view projects/*/.claude/skills/ # generated or symlinked skill view projects/*/agentstack.lock # generated during demo runs runtime/ # simulated machine state ~/.codex, ~/.claude, ~/.pi # provider-owned private state
Place Purpose Commit?
fixtures/central-library Demo input files used to seed the central library. Yes, for the sandbox demo.
~/.agentstack/lib Actual managed home for installed skills and server definitions. No, machine-local state.
projects/*/agentstack.toml Project source of truth: profiles reference names. Yes.
.mcp.json, .claude/skills Provider-readable generated views. No.

Quick start.

Add a skill and MCP server to the central library, then reference both by name from a profile.

central-library flow 
agentstack lib add-server kibana --file fixtures/central-library/kibana.toml --write
agentstack lib add sql-review --path fixtures/central-library/sql-review --write
agentstack lib list

# projects/central-demo/agentstack.toml
[profiles.central]
servers = ["kibana"]
skills = ["sql-review"]

agentstack --manifest-dir projects/central-demo use central --write
agentstack --manifest-dir projects/central-demo explain kibana
agentstack --manifest-dir projects/central-demo doctor

Existing projects.

Old projects do not need a flag day. Inline definitions and local skills still work, and they can coexist with central references.

Before: repo-local definitions

This is still valid. It is the compatibility mode many existing projects start with.

old project agentstack.toml 
[servers.kibana]
type = "http"
url = "https://kibana.example.com/mcp"

[servers.kibana.headers]
Authorization = "Bearer ${KIBANA_TOKEN}"

[skills.sql-review]
path = "./skills/sql-review"

[profiles.backend]
servers = ["kibana"]
skills = ["sql-review"]

After: central references

The project keeps only the selection. Definitions live in ~/.agentstack/lib.

centralized agentstack.toml 
[profiles.backend]
servers = ["kibana"]
skills = ["sql-review"]

Migrate a local skill

one skill at a time 
agentstack lib add sql-review --path ./skills/sql-review
agentstack lib add sql-review --path ./skills/sql-review --write

# Then remove the local [skills.sql-review] table from the project.
# The existing profile name keeps working:
# skills = ["sql-review"]

Migrate an inline server

Today this is a manual export to a server file, then lib add-server. A future --from-manifest helper can automate it.

server definition file 
# kibana.toml
type = "http"
url = "https://kibana.example.com/mcp"

[headers]
Authorization = "Bearer ${KIBANA_TOKEN}"

agentstack lib add-server kibana --file kibana.toml --write

# Then remove the local [servers.kibana] table from the project.
# servers = ["kibana"] now resolves from the library.
Inline override is intentional: if a project keeps [servers.kibana] or [skills.sql-review], that local definition wins over the central library. Remove the inline table only when you want the central copy to be used.

New projects.

For a fresh project, start central. Add capabilities to the library once, then keep the project manifest small.

Recommended project shape

my-project/ agentstack.toml # profiles and targets only src/ README.md

No copied skill folder and no inline MCP definition are required unless this project needs an override.

New-project setup

fresh project 
agentstack lib list
agentstack lib add-server kibana --file ~/agentstack-fixtures/kibana.toml --write
agentstack lib add sql-review --path ~/agentstack-fixtures/sql-review --write

cat > agentstack.toml <<'EOF'
version = 1

[profiles.backend]
servers = ["kibana"]
skills = ["sql-review"]

[targets]
default = ["claude-code", "codex"]
EOF

agentstack use backend --write
agentstack doctor

Secrets and locking.

The central library stores definitions, never machine-local secret values.

Safe central definition

${REF} stays in the library 
[headers]
Authorization = "Bearer ${KIBANA_TOKEN}"

At render or gateway time, the local resolver turns ${KIBANA_TOKEN} into the machine's actual value.

Never commit this in the library

plaintext value 
[headers]
Authorization = "Bearer real-token-value"

lib add-server warns on suspicious literal secrets. The design rule is still simple: central files hold refs, not values.

Skill content is also scanned on install: hidden Unicode (zero-width characters, bidi overrides) blocks the install, prompt-injection heuristics warn, and agentstack audit re-scans everything materialized — so what enters the library is checked, not just checksummed.

Item Locked material Not locked
Skill Content checksum and source locator. Generated provider copies.
MCP server Definition checksum with ${REF} placeholders. Resolved secret values and provider-specific render shape.

Generated provider views.

CLIs still need files in the shapes they understand. AgentStack renders those views from the central source.

Provider view example

simulated ~/.claude.json after use --write 
{
  "mcpServers": {
    "kibana": {
      "type": "http",
      "url": "https://kibana.example.com/mcp",
      "headers": {
        "Authorization": "Bearer demo-secret-value"
      }
    }
  }
}

Why this is okay

  • The provider view is generated output.
  • It is backed up before mutation.
  • Only AgentStack-owned entries are changed or pruned.
  • The central library remains the source of truth.
