Strategy · detailed public plan · 16 July 2026

Give every agent only what it needs—and prove what it did.

AgentStack is becoming the vendor-neutral trust, policy, and workflow control plane for agent-enabled repositories. Portability creates adoption. Enforced grants and evidence create differentiation. Team coordination creates revenue.

Contract

Specify before composing boundaries

Define agentstack run <harness> --locked, the no-Docker protected tier, the Docker maximum-assurance tier, trust timing, lock guarantees, secret handling, evidence, and cleanup.

Build

One path from trust to report

Resolve only locked inputs, compile repository policy under the machine ceiling, launch through the canonical gateway, record material decisions, and fail before activation when trust or reproducibility cannot be proven.

Prove

Four demos and unassisted activation

Prove a safe run, policy denial, changed or missing locked input, and a separate maximum-assurance Docker run. Add witnesses that authoritative verification hashes current bytes, every D2 delivery path uses one grant, and a one-byte D3 edit re-gates trust.

After both Phase 0 gates

Phase 1 — productize the wedge and exact Workspace Grants.

DIRECTORY BOUNDARY

Read, write, deny

Start with safely mountable directory roots, keep the repository root read-only, reject traversal and symlink escapes, and never describe a full read-only repository mount as scoped read access.

ROLE BOUNDARY

Different authority for each job

Bind grants to profiles and future roles: researcher, implementer, reviewer, and deployer receive only the folders, tools, secrets, egress, commands, and budgets they need.

PORTABLE PLAN

Freeze semantics once

Version one backend-neutral execution plan. A local or hosted adapter may narrow it but cannot reinterpret or widen it.

EXIT EVIDENCE

Real use, not only tests

Ten activated repositories, five weekly users, three CI users, one dependent team, and an external run proving out-of-grant writes fail.

D5 POSTURE

Name the network boundary honestly

Use compatibility evidence to decide whether ordinary sandboxing becomes topologically confined by default or the weaker bridge/proxy path receives a clearly weaker name.

EVIDENCE HEURISTIC

Flag suspicious sequences, never claim DLP

After recorder completion, report a secret access followed shortly by egress inconsistent with that secret’s constrained server. It is metadata correlation only: report, do not block or inspect payloads.

Exit gate

  • Ten independent repositories activate; five external users return weekly for four consecutive weeks.
  • Three repositories use the GitHub Action and one external team relies on AgentStack for real work.
  • An external run proves writes outside its grant fail; the frozen plan passes local backend conformance tests.

After Phase 1 evidence

Phase 2 — paid partners, then Cloudflare, then workflows.

First

Find the repeated paid problem

Run limited design partnerships around a governed rollout, policy baseline, or evidence outcome. Avoid unlimited bespoke engineering.

Second

Cloudflare one-shot runner

A Worker owns authentication, policy, approval, and credential brokering; one isolated Sandbox handles one repository, harness, task, and frozen grant, then returns a validated patch and receipt. The sandbox receives approved read material and short-lived execution credentials—not a long-lived GitHub or model credential. Trusted code validates every returned path, traversal, symlink, and write before any approved branch or pull request. Start bring-your-own; no multi-tenant dashboard yet.

Third

Saved governed workflows

Only after local/cloud parity and a real repeated task. Import a native format first where practical, normalize a restricted JS/TS API to the frozen plan, grant each role separate authority, digest-pin source and normalized plans, and re-gate changes. Retries require idempotency; irreversible effects require policy or approval. Add Cloudflare Workflows only when proven waits, schedules, retries, recovery, or approval events need durability.

Exit gate

  • Two paying partners share the same coordination problem.
  • One partner completes a Cloudflare run whose receipt matches the local grant contract.
  • One saved, role-scoped workflow repeats without permission widening.
  • Buyer, daily user, and security stakeholder agree on the value.

Ecosystem architecture

One coherent workspace now; independent products only when earned.

The TanStack analogy describes the brand architecture, not a repo-count target. Present Core, Trust, Policy, Runtime, Workflow, Recorder, Cloud, and CLI as recognizable product layers while retaining the nine-crate monorepo.

ORGANIZATION TRIGGER

Transfer when ownership has value

Reserve the organization name. Transfer the main repository when a second public repository is genuinely needed, a partner needs organization ownership, an external maintainer needs durable permissions, or neutral ownership materially improves trust.

SPLIT TEST

Five tests before a new repository

The component must work without the full CLI, have its own users/docs/versioning/release cadence, expose a stable API, reduce more coordination than it adds, and have a real adopter asking to use or contribute independently.

Revenue boundary

Keep local enforcement open. Charge for coordination.

OPEN SOURCE

Portable local foundation

Manifest, lock, adapters, trust, policy, Workspace Grants, local execution, signatures, reports, and CI verification stay inspectable and adoptable.

DESIGN PARTNERS

First revenue

Sell a bounded rollout or evidence outcome and use repeated manual work to discover the team product.

TEAM

Coordination layer

Shared policy and workflows, identity, approvals, evidence, managed updates, exceptions, and optional hosted execution.

ENTERPRISE

Assurance and operations

Private deployment, residency, SSO, governance integrations, retention, compliance evidence, onboarding, and support.

Commercial rule: the first paid offer is a bounded governed rollout, policy baseline, migration, or evidence outcome with a defined success measure. Exact pricing remains a private hypothesis tested in sales conversations. Repeated manual work becomes product input; one customer’s bespoke procurement list does not become the roadmap.

Metrics and stop signals

Measure behavior, not roadmap volume.

ACTIVATION

Can people reach value?

Quickstart completion, median time to protected run, failure stage, and external repositories with committed manifests and lockfiles.

RETENTION

Does it become routine?

Weekly external users and repositories, repeat runs, enforced scoped writes, saved workflow reruns, local/cloud plan parity, CI checks, and multi-client use.

TRUST VALUE

Did the boundary matter?

Violations blocked before activation, drift detected in CI, time to explain allow/deny decisions, and evidence exported to code or security review.

COMMERCIAL PULL

Will an organization pay?

Trial requests, teams offering a real repository and policy baseline, qualified paid pilots, and repeated demand for shared policy, identity, inventory, workflows, or evidence.

Change the wedge if adequate distribution shows that repository configuration is not a meaningful trust problem, routine use has too much friction, native standards solve portability plus machine ceilings, or interest is only generic MCP discovery. Strong portability with weak trust suggests a narrower configuration product; strong security-team pull with weak developer pull suggests a CI and platform-governance lead.

Risks, limits, and documentation governance

Do not let a compelling future weaken today’s contract.

  • CLAIMSDistinguish cooperative host controls from kernel-enforced sandbox and lockdown guarantees.
  • CLOUDCloudflare is an adapter behind the portable frozen plan; begin with bring-your-own cloud and model credentials.
  • WORKFLOWSJS/TS workflow files request authority but never grant it, run arbitrary code on the host, or bypass digest re-gating.
  • CREDENTIALSKeep GitHub, model, and long-lived credentials in trusted broker code, not the task sandbox.
  • FOCUSDefer marketplace, broad fleet platform, compliance suite, additional backends, and new client adapters until real demand earns them.
  • POSITIONOwn enforcement-backed action evidence, not generic observability, traces, token analytics, evaluations, DLP, or payload inspection.
  • MARGINSEnforce time, token, and cost budgets before reselling hosted execution.
  • PILOTSTime-box design partnerships and productize only repeated coordination needs.
  • DOC STATUSLabel material Stable, Experimental, Design, Historical, or Archived; label examples as tested, Docker reproductions, validated manifests, or illustrative.
  • SYNCGenerate this public view from authoritative Markdown or enforce a synchronization test for phases, status, examples, versions, adapter count, and claims.
